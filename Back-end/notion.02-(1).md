# 객체지향
## 1. 객체지향이 무엇인가?
(**O**bject-**O**riented **P**rogramming)   

**객체지향이란?** 데이터와 기능을 하나로 묶은 객체 중심으로 프로그램을 만드는 방식  
즉, 객체들이 서로 메시지를 주고받으며 상호작용하도록 설계하는 방식

**왜 사용함?**    
- **유지보수가 쉬움**   
기능을 객체별로 분리해서 문제 생긴 부분만 수정 가능
- **재사용 가능**   
한 번 만든 객체를 여러 곳에서 다시 사용 가능
- **코드가 직관적임**   
현실 구조와 비슷해서 이해 쉬움

### 1-1. 기본 구조
**클래스 -> 객체 -> 메서드 호출**

**클래스(Class)**  
객체를 만들기 위한 설계도 
객체가 가져야 할 데이터와 기능을 미리 정의  
(즉, 객체를 만들기 위한 틀)

**객체(Object)**  
클래스를 기반으로 만들어진 실제 존재하는 데이터  
(즉, 클래스의 실제 결과물)

**속성(Attribute)**  
객체가 가지고 있는 정보나 상태값  
-> 객체마다 다른 값을 가질 수 있음  
(즉, 객체의 정보)

**메서드(Method)**  
객체가 수행할 수 있는 동작이나 기능
(즉, 객체의 행동)

**생성자(Constructor)**   
객체가 생성될 때 자동으로 실행되는 특별한 메서드  
-> 객체를 만들 때 초기값 설정할 때 사용     
(즉, 객체 생성 초기 설정)

**인스턴스(Instance)**   
특정 클래스로부터 생성되어 소속된 객체를 의미

### 1-2. 내부 과정
**메모리 할당**   
객체가 만들어질 때 일어나는 과정
- 객체는 Heap 영역에 생성
- 생성 시 클래스 구조에 맞게 필드 공간 확보
- 각 객체는 독립된 메모리 공간 가짐
  -> 즉, 같은 클래스라도 객체마다 데이터는 따로 저장됨

**참조 관리**   
변수가 객체를 직접 가지는 게 아닌 주소를 가짐
- Heap에 있는 객체의 주소 저장
- 여러 변수가 하나이ㅡ 객체를 공유할 수도 있음
- 객체는 직접 접근이 아니라 참조를 통해 접근

**메서드 호출**   
객체를 통해 간접적으로 실행  
- 메서드는 객체에 연결된 형태로 호출
- 실제 실행 시, 해당 객체의 메서드를 찾아서 실행

## 2. 객체지향의 핵심 개념
**캡슐화(Encapsulation)**   
데이터와 메서드를 하나로 묶고 외부에서 직접 접근하지 못하게 보호하는 것   
**왜 사용할까?** 객체 내부 데이터를 안전하게 관리하기 위해 사용

**상속(Inheritance)**   
기존 클래스의 속성과 기능을 새로운 클래스가 물려받는 것   
**왜 사용할까?** 코드 재사용이 가능

**다형성(Polymorphism)**   
하나의 타입으로 여러 객체를 동일하게 다루지만, 실행 결과는 다르게 나오는 것   
(오버로딩 / 오버라이딩)   
**왜 사용할까?** 같은 명령으로 다양한 객체 처리 가능

**추상화(Abstraction)**  
복잡한 내부는 숨기고 필요한 기능만 보여주는 것    
**왜 사용할까?** 사용하기 쉽게 만들기 위해

### 2-1. 오버로딩
**오버로딩(Overloading)이란?** 같은 이름의 메서드를 매개변수만 다르게 여러 개 만드는 것   
(즉, 같은 기능 이름인데 입력값에 따라 다른 메서드 실행됨)

**특징**    
- 메서드 이름 같음
- 매개변수 달라야 함
- 같은 클래스 안 가능

### 2-2. 오버라이딩
**오버라이딩(Overriding)이란?** 부모 클래스의 메서드를 자식 클래스에서 재정의해서 다르게 만드는 것

**특징**  
- 이름 같음
- 매개변수 같음
- 상속 관계 필요
- 내용만 변경

### 2-3. 오버로딩과 오버라이딩의 차이점
| 구분 | 오버로딩 | 오버라이딩 |
| ----- | ----- | ----- |
| 의미 | 같은 이름 다른 매개변수 | 부모 메서드 재정의 |
| 위치 | 같은 클래스 | 상속 관계 |
| 목적 | 편의성 | 다형성 구현 |


## 3. 객체지향의 장점
**코드 재사용성이 높음**  
상속을 통해 이미 만들어 놓는 클래스를 다시 사용할 수 있음   
(비슷한 기능을 가진 클래스를 새로 만들 때 처음부터 작성하지 않아도 됨)  
- 수정 시 여러 클래스에 자동 반영 가능
- 코드 중복 감소
- 개발 시간 단축
- 오류 감소 

**유지보수 및 확장이 쉬움**  
객체별로 나누어져 있어서 문제가 생겼을 때 수정 위치를 쉽게 찾을 수 있음     
(관련 코드가 한 클래스 안에 모여 있음)
- 오류 수정 및 기능 추가 쉬움
- 코드 관리 편리
- 수정 범위 파악 쉬움
 
기존 코드를 크게 수정하지 않고 새 클래스를 추가      
-> 큰 프로그램에 유리
- 구조 유지 쉬움
- 수정 범위 최소화

**현실 세계 표현이 자연스러움**  
객체지향은 현실의 사물을 그대로 코드로 표현하기 좋음  
(즉, 생각한 구조 그대로 코드로 옮기기 쉬움)  
-> 사람이 프로그램 구조를 이해하기 쉬워짐   
- 요구사항 반영 쉬움
- 설계와 구현이 쉬움
- 직관적인 코드 작성 가능

**협업이 편함**  
객체별로 역할 분리  
-> 각자 자기 객체만 개발하면 되기 때문에 서로의 간섭이 줄어듦
- 충돌 감소     
- 대규모 개발 유리
- 코드 병합 부담 감소
- 책임 구분 명확

**보안성 높음**  
캡슐화를 통해 중요한 데이터를 외부에서 직접 변경하지 못하게 만들 수 있음    
(메서드를 통해서만 변경 가능)   
-> 데이터 보호에 유리   
- 내부 구현 숨김 가능
- 무결성 유지 가능
- 허용된 방식만 변경 가능


## 4. 객체지향의 단점
**설계 복잡**  
코드를 바로 작성하는 것이 아니라 전체 구조를 먼저 설계해야함     
(초반 설계 어려움)
- 어떤 객체가 필요한지 먼저 정의
- 객체 간 객체를 먼저 결정해야 함

**작은 프로그램 비효율적**  
간단한 프로그램인데도 클래스, 객체, 메서드를 만들어야 해서 코드가 길어짐    
- 간단한 기능도 클래스, 객체, 메서들 나눠야 함
- 코드 양이 늘어나고 구족 복잡해짐

**실행속도가 느릴 수 있음**  
추가적인 구조 때문에 성능 오버헤드 생김  
- 객체 생성 과정에서 메모리 할당 발생
- 참조를 통해 접근하므로 절차지향보다 느릴 수 있음
객체 생성과 메서드 호출 과정이 추가되므로 절차지향보다 느려질 수 있음   
-> 성능이 중요한 경우 단점이 될 수 있음     

**메모리 사용량 많음**  
객체 단위로 데이터를 관리하기 때문에 메모리를 더 사용
- 각 객체마다 속성 저장 공간이 필요
- 메서드 호출을 위한 구조도 함께 사용  
  -> 결과적으로 메모리 사용량이 증가
