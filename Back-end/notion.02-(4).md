# 무상태성
## 1. 무상태성에 대해
**무상태성(Stateless)란?** 서버가 이전 클라이언트의 요청 상태를 기억하지 않는 구조  
(즉, 클라이언트가 보낸 각 요청은 서버 입장에서 완전히 새로운 요청으로 처리됨)

**상태(State)란?** 이전 요청에서 생긴 정보

## 2. 상태 저장 방식과 비교
**상태 저장(Stateful)이란?** 서버가 사용자 상태를 기억      
(서버 메모리 안에 저장됨)
-  서버가 사용자를 기억함
- 매 요청마다 인증 정보가 필요하지 않을 수 있음
- 사용자별 상태 관리 가능

**무상태 방식(Stateless)란?** 서버가 이전 상태를 저장하지 않는 방식     
- 서버가 사용자 기억 안 함
- 각 요청이 독립적
- 필요한 정보는 요청마다 포함

### 2-1. 비교 표
| 구분 | 상태 저장 방식 | 무상태 방식 |
| ----- | ----- | ----- |
| 서버 기억 | 있음 | 없음 |
| 요청 독립성 | 낮음 | 높음 |
| 서버 부담 | 큼 | 적음 |
| 확장성 | 낮음 | 높음 |
| 인증 처리 | 간단 | 복잡 |


## 3. REST에서 왜 중요할까?
REST는 각 요청을 독립적으로 만들기 위해서 무상태성을 사용   
(즉, 하나의 요청만 봐도 서버가 처리 가능해야 함)

### 3-1. REST에서 얻는 효과
**서버 독립성**     
각 요청이 독립적이므로 어느 서버든 같은 방식으로 처리 가능

**확장성 향상**     
사용자가 많아지면 서버를 쉽게 추가 가능

**유지보수 쉬움**   
서버가 사용자 상태를 저장하지 않음  
-> 관리해야 할 정보가 줄어듦

**안정성 증가**     
특정 서버가 고장나도 다른 서버가 요청 처리 가능

## 4. 실제 동작 과정
무상태 구조에서는 서버가 로그인 상태를 기억하지 않음    
-> 클라이언트가 매 요청마다 인증 정보를 함께 보내야 함

**로그인 요청**     
사용자가 아이디와 비밀번호를 서버에 전송함

**서버 인증 처리**      
서버는 사용자의 정보를 확인함   
인증 성공 -> 사용자를 식별할 수 있는 토큰 생성

**토큰 발급**   
서버는 인증 후 토큰을 응답함    
-> 토큰이 사용자의 로그인 상태를 대신하는 값

**클라이언트 저장**     
클라이언트는 토큰을 저장함

**다음 요청 시 토큰 전송**  
클라이언트는 새로운 요청마다 토큰을 함께 보냄

**서버가 다시 확인**    
서버는 이전 요청을 기억하지 않음    
-> 현재 요청 안의 토큰만 검사함

**응답 반환**   
정상이라면 서버가 데이터를 반환함

## 5. 왜 서버가 상태를 저장하지 않을까?
더 단순하고 확장하기 쉬운 시스템을 만들기 위함

**서버 구조를 단순하게 만들기 위해**    
서버가 상태를 저장하면 사용자마다 정보를 계속 기억해야 함   
- 사용자가 많아질수록 저장해야 하는 정보 늘어남
- 상태를 저장하지 않으면 서버는 요청 처리만 담당하면 됨

**메모리 사용을 줄이기 위해**   
상태를 저장하면 서버 메모리를 계속 사용함
- 사용자가 많아질수록 메모리 부담 증가
- 무상태 구조는 메모리 부다 감소

**장애 대응을 쉽게 하기 위해**  
특정 서버가 고장나도 다른 서버가 요청 처리 가능     

## 6. 장점
**서버 구조가 단순해짐**    
서버가 사용자 상태를 저장하지 않기 때문에 처리해야 하는 역할이 줄어듦   
- 코드 단순함
- 관리 쉬움
- 오류 감소

**서버 메모리 부담 감소**   
상태를 저장하지 않으므로 사용자별 메모리 점유가 줄어듦  
-> 메모리 효율 좋아짐

**장애 복구 쉬움**  
특정 서버가 고장나도 다른 서버가 요청 처리 가능     
(서버에 상태가 없기 때문)
- 서비스 안정성 향상

**캐시 사용이 쉬움**    
요청이 독립적이라 응답 캐싱이 시움
- 속도 향상
- 서버 부하 감소

## 7. 단점
**인증 처리 복잡**  
서버가 로그인 상태를 기억하지 않으므로 매 요청마다 인증 필요
- 토큰 관리 필요
- 보안 처리 복잡

**요청 데이터가 커질 수 있음**  
매 요청마다 필요한 정보 전송
- 네트워크 사용 증가
- 성능 저하 가능

**클라이언트 부담 증가**    
상태 일부를 클라이언트가 관리해야 함    

**반복 처리 증가**  
서버는 매 요청마다 같은 인증 과정 반복
(CPU 사용 증가 가능)

**복잡한 사용자 흐름 처리 어려움**  
사용자 진행 상태를 직접 관리하지 않기 때문에 구현 복잡해 질 수 있음
