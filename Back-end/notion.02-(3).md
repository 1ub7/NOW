# RESTful API 
## 1. API
(**A**pplication **P**rogramming **I**nterface)     
**API란?** 프로그램끼리 기능이나 데이터를 주고받기 위한 규칙    
(대부분의 API는 Stateless 구조)

**역할**    
내부 구현을 숨기고, 정해진 인터페이스를 통해 기능을 외부에 제공하는 것
- 필요한 기능만 제공
- 서버 기능을 외부에서 호출 가능
- 개발자 간 협업을 쉽게 함 
- 시스템 간 결합도를 낮춤

**구조**    
기본적으로 요청-응답 구조로 동작  
(주로 HTTP 기반으로 통신)  
- 클라이언트가 요청 전송
- 서버가 요청 처리
- 서버가 결과를 응답으로 반환

## 2. REST
(**Re**presentational **S**tate **T**ransfer)   
**REST란?** 
자원을 URI로 표현하고, HTTP 메서드를 사용해 자원을 처리하는 아키텍처 스타일     
(자원을 중심으로 설계하는 API 방식)

### 2-1. REST 특징
**자원(Resource) 중심 구조**    
기능이 아닌 데이터(자원) 중심으로 설계
- API 구조 직관적임
- 어떤 데이터인지 URL만 보고 이해 가능

**HTTP 메서드로 의미 구분**     
URL 하나에 기능을 넣지 않고, HTTP 메서드로 행동을 나눔
- URL 깔끔
- 코드 구조 단순화
- 같은 자원도 다양한 동작 가능

**무상태(Stateless)**   
서버는 이전 요청을 저장하지 않음
- 서버 구조 단순
- 여러 서버로 확장 쉬움

**URL은 명사 중심**     
기능이 아닌 자원을 표현하기 때문
- API 구조 표준화
- URL만 봐도 무엇을 다루는지 알 수 있음
- 유지보수 쉬워짐

## 3. RESTful API
**RESTful API란?** REST 설계 원칙을 잘 지켜서 만든 API를 의미   
(즉, REST 원칙을 준수하여 설계된 웹 API)

### 3-1. 핵심 구조
**자원(Resource)**  
모든 데이터는 자원으로 표현됨 (URI로 자원 구분)     
- /users -> 사용자 전체
- /users/1 -> 특정 사용자

**행위(HTTP Method)**   
URL이 아니라 HTTP 메서드가 기능을 결정  
(02-(2)-4번 참고)

**표현(Representation)**    
자원의 상태를 클라이언트와 서버 간에 주고받는 데이터 형태
- 동일 자원이라도 다양한 형태 가능  
(JSON, XML 등)
- 데이터 구조를 클라이언트에 전달

위에 나온 **JSON이란?**         
(**J**ava**S**cript **O**bject **N**otation)        
데이터를 구조적으로 표현하기 위한 텍스트 기반 포맷  
(서버와 클라이언트가 데이터를 주고받는 표준 형식)

그 옆에 있는 **XML이란?**   
(e**X**tensible **M**arkup **L**anguage)    
데이터를 구조적으로 표현하고 전달하기 위한 마크업 언어  
(태그를 이용해서 데이터를 계층 구조로 표현하는 방식)

### 3-2. RESTful API 특징
**URL은 명사 중심**     
기능이 아니라 데이터 중심으로 설계  
**왜 그럴까?** URL은 자원, 행동은 HTTP 메서드가 담당
- URL만 보고도 어떤 데이터를 다루는지 알 수 있음
- API 구조가 직관적이고 일관적임

**무상태(Stateless)**   
서버는 이전 요청을 저장하지 않음    
-> 모든 요청은 독립적으로 처리됨    
**왜 이렇게 설계함?** 서버가 상태를 저장하면 관리가 복잡해짐
- 서버 구조 단순
- 확장성 높음 (서버 여러 개 가능)

**일관된 인터페이스**   
모든 API가 같은 규칙을 따름     
-> 구조가 통일되어 이해하기 쉬움    
**왜 중요함?** 모든 API 사용 방식이 비슷해져서 새로운 API도 쉽게 이해 가능

**HTTP 표준 활용**  
기존 웹 기술 그대로 사용    
(브라우저/모바일/서버 모두 호환 가능)   
**왜 사용함?** 기존 인터넷 구조를 그대로 사용할 수 있음     
-> 플랫폼 간 호환성이 매우 높음

## 4. REST와 RESTful의 장단점
### 4-1. REST 장점
**구조가 단순하고 이해하기 쉬움**   
복잡한 함수 호출 대신 웹 규칙만 사용    
- URL + HTTP 메서드 조합이라 직관적
- 규칙이 일정해서 학습 난이도 낮음

**확장성 좋음**     
서버가 상태를 기억하지 않기 때문
- Stateless 구조 덕분에 서버 여러 개로 확장 가능  
- 트래픽 증가에도 대응 쉬움

**프론트/백엔드 분리 가능**     
개발 속도 증가 + 역할 분리 가능
- 프론트 -> API만 호출
- 백엔드 -> 데이터만 제공

### 4-2. REST 단점
**무상태(Stateless)로 인한 비효율**     
서버가 상태를 기억하지 않음 
- 매 요청마다 인증 정보 포함 필요
- 같은 정보도 반복 전송해야 함

**복잡한 기능 처리 어려움**     
CRUD 중심 구조 -> REST만으로는 설계 불편
- 실시간 채팅
- 복잡한 트랜잭션
- 다중 단계 처리

**설계 품질 차이 발생**  
REST는 강제표준이 아님
- 개발자마다 다르게 설계 가능
- REST처럼 보이는 REST가 아닌 API 존재

### 4-3. RESTful API 장점
**구조가 단순하고 이해하기 쉬움**   
구조가 직관적임
- 처음 봐도 어떤 기능인지 쉽게 파악 가능
- 협업 시 의사소통 쉬움
- API 문서를 이해하기 쉬움

**유지보수가 쉬움**     
설계 규칙이 일정해서 수정과 관리가 편함
- URL 규칙이 통일됨
- 기능이 메서드로 구분됨
- 코드 구조가 예측 가능함
- 기능 추가 시 기존 구조 유지 가능

**확장성 높음**     
무상태 구조라 서버 확장 쉬움
- 서버가 사용자 상태를 기억하지 않음
- 요청만 처리하면 됨
- 서버 여러 대로 분산 가능  
    -> 사용자 수 증가 시 대응 쉬움
- 로드 밸런싱 적용 쉬움

**플랫폼 독립적**   
같은 API를 여러 환경에서 사용할 수 있음     
-> 하나의 서버로 여러 클라이언트 지원 가능

### 4-4. RESTful API 단점
**상태 관리 어려움**    
무상태 구조 때문에 서버가 로그인 상태를 기억하지 않음   
- 매 요청마다 인증 정보 필요
- 세션 관리가 복잡해질 수 있음
->  인증 구현이 복잡해질 수 있음

**요청 횟수가 많아질 수 있음**  
복잡한 데이터를 얻으려면 여러 번 요청해야 함    
- 네트워크 부하 증가
- 속도 저하 가능
- 모바일 환경에서 불리

**과도한 데이터 전송 가능**     
필요한 데이터보다 더 많이 받을 수 있음  
- 불필요한 데이터 전송 가능
- 성능 비효율 발생

## 5. REST와 RESTful 차이점
| 구분 | REST | RESTful |
| ----- | ----- | ----- |
| **의미** | 웹 자원을 관리하기 위한 설계 원칙 | REST 원칙을 지켜 만든 실제 API |
| **형태** | 개념적인 아키텍처 스타일 | 구현된 서비스 형태 |
| **역할** | API를 어떻게 설계할지 기준 제공 | REST 방식대로 동작하는 API |
| **초점** | 이론 / 규칙 중심 | 구현 / 적용 중심 |
| **관계** | 기본 규칙 | REST를 따른 결과물 |
