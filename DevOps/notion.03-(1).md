# 이미지(Image)
## 1. 정의
**이미지란?**    
컨테이너를 만들기 위한 설계도   
(컨테이너를 실행하기 전에 미리 준비된 실행 환경 묶음)

**왜 중요함?**      
환경 차이 문제를 없애기 때문    

**왜 필요함?**      
컨테이너를 만들기 위해

## 2. 역할
컨테이너를 만들기 위한 실행 환경 템플릿 역할    
(컨테이너를 실행하기 전에 준비된 설계도)

**실행 환경 제공**      
프로그램이 실행되기 위한 환경을 담음    
그래서 어느 컨퓨터에서도 같은 환경 실행 가능    

**컨테이너 생성 기준**  
컨테이너는 이미지를 기반으로 만들어짐
즉, 이미지가 없으면 컨테이너 생성 불가

**버전 관리 가능**      
이미지는 버전별 저장 가능   
(이전 상태 복구 가능)

**배포 단순화**     
이미지 하나만 전달하면 됨   
(배포가 쉬워짐)

## 3. 이미지 생성 방법
보통 Dockerfile로 만듦   
### 3-1. Dockerfile 기본 구조
ex
```
FROM python:3.12
WORKDIR /app
COPY . .
RUN pip install -r requirements.txt
EXPOSE 5000
ENV APP_ENV=production
CMD ["python", "app.py"]
```
한 줄씩 문장의 역할을 정의하면?
```
기반 이미지 선택
↓
작업 폴더 지정
↓
파일 복사
↓
필요한 설치
↓
포트 지정
↓
환경 변수 설정
↓
실행 명령 지정
```

### 3-2. Dockerfile 명령어
**FROM**    
이미지를 만들 때 어떤 운영 환경을 사용할지 정함     
Docker 이미지는 반드시!! 하나의 기반 이미지 위에서 시작     
-> 운영체제나 언어 런타임이 포함된 이미지를 선택

**WORKDIR**     
컨테이너 내부에서 앞으로 작업할 폴더 설정   
이후 명령어는 해당 경로를 기준으로 실행
- 경로를 반복해서 쓰지 않아도 됨
- 파일 관리가 편함
- 실행 위치 통일

등의 장점이 있음

**COPY**    
현재 폴더 파일을 이미지 내부로 복사     
프로그램 코드나 설정 파일을 컨테이너 안으로 넣을 때 사용    

**RUN**     
이미지 생성 중 필요한 명령을 실행       
컨테이너가 실행될 때가 아니라 빌드할 때 실행됨  
주로 하는 일
- 패키지 설치
- 라이브러리 설치
- 업데이트
- 설정 변경

**EXPOSE**  
컨테이너가 사용할 포트 명시     
외부에서 어떤 포트로 접근할지 알려줌
주의할점!!! 포트를 여는게 아니라 포트 정보를 명시하는 역할

**ENV**     
컨테이너 안에서 사용할 환경 변수 저장    
프로그램 설정값을 코드 밖에서 관리할 수 있음    
(프로그램은 이걸 읽어 동작 변경 가능)

**CMD**     
컨테이너 시작 시 실행할 기본 명령   
(즉, 프로그램 시작 담당)    
-> CMD는 기본값이라서 실행 시 변경 가능

## 4. 계층 구조
┌─────────────────────┐
│ App Source          │
├─────────────────────┤
│ Python              │
├─────────────────────┤
│ Libraries           │
├─────────────────────┤
│ Ubuntu Base         │
└─────────────────────┘     
**왜 계층 구조를 씀?**  
- 저장 공간 절약    
    공통 레이어 재사용      
    중복 저장 안 함

- 빌드 속도 향상    
    변경된 부분만 다시 생성     
    (코드만 수정할 시 더 빠름)

- 효율적 관리
    레이어 단위 캐시 가능   
    빌드 최적화 가능

## 6. 이미지 주요 명령어
### 6-1. 다운로드
**docker pull** : 저장소에서 이미지를 내려받음  
- 로컬 PC에 저장됨
- 실행 전에 미리 받아둘 수 있음
- 이미지가 여러 레이어로 나뉘어 순서대로 다운로드됨
- 이미 같은 레이어가 있으면 다시 받지 않음

### 6-2. 생성
**docker build** : Dockerfile을 이용해 이미지를 생성함  
- 현재 폴더의 Dockerfile 읽음   
- 새로운 이미지 생성
- 각 명령어마다 새로운 레이어 생성
- 이전 레이어를 재사용하여 빌드 속도 향상 가능

### 6-3. 확인
**docker images** : 로컬에 저장된 이미지 목록 확인  
- 이미지 상세 정보
- 레이어 구조
- 생성 시간     

**docker inspect** : 이미지 상세 정보 확인 가능
- 레이어 구조 확인
- 메타데이터 확인 가능

### 6-4. 삭제
**docker rmi** : 이미지 삭제       
- 컨테이너에서 사용 중이면 삭제 안 될 수 있음
- 저장 공간 확보 가능
- 사용하지 않는 이미지 정리 가능

### 6-5. 업로드
**docker push** : 로컬 이미지를 Docker Hub 등에 올림    
- 다른 컴퓨터에서도 사용 가능하게 저장소에 등록
- 팀원과 동일한 이미지 공유 가능
- 버전 관리 가능
- 배포 자동화에 사용 가능

## 7. 특징
**읽기 전용**   
기본적으로 수정되지 않는 읽기 전용 파일
- 이미지를 직접 변경하지 않음
- 수정하면 새 이미지 생성
- 원본 그대로 유지

**계층 구조**   
여러 개의 레이어로 구성됨
- 변경된 부분만 다시 생성
- 공통 레이어 재사용 가능

**불변성**  
한 번 생성된 이미지는 변하지 않음
- 같은 이미지는 항상 같은 환경 제공
- 실행 결과 예측 가능
- 오류 재현 쉬움

**이식성**     
어디서든 같은 이미지 실행 가능

**공유 가능**       
이미지는 저장소를 통해 쉽게 공유 가능
- 개발자마다 같은 환경 사용 가능
- 배포 자동화에 활용 가능

## 8. 장점
**환경 일관성**     
개발 환경과 운영 환경이 같아짐      
-> 환경 차이 문제 감소

**빠른 배포 가능**      
이미 만들어진 이미지를 바로 실행 가능   
-> 설치 과정 없이 즉시 실행

**재사용 가능**     
하나의 이미지를 여러 컨테이너에서 사용 가능     
-> 효율적 관리 가능

**저장 공간 절약**      
레이어 재사용 가능      
-> 디스크 낭비 감소

## 9. 단점
**이미지 크기가 커질 수 있음**  
불필요한 파일이 포함되면 이미지가 커짐  
- 다운로드 느려짐
- 저장 공간 증가

**보안 취약점 포함 가능**   
이미지 안에 취약한 패키지가 있을 수 있음    
- 보안 위험 발생 가능

**관리 복잡**   
이미지가 많아지면 관리 어려움   
- 버전 혼란 가능

**빌드 시간 증가**  
이미지가 복잡하면 생성 시간이 오래 걸림     
- 개발 속도 저하 가능
