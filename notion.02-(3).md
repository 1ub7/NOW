# RESTful API
## 1. API
(**A**pplication **P**rogramming **I**nterface)     
**API란?** 프로그램끼리 기능이나 데이터를 주고받기 위한 인터페이스

**역할**    
내부 구조를 직접 알지 않아도 기능을 사용할 수 있게 해줌
- 필요한 기능만 제공
- 서버 기능을 외부에서 호출 가능
- 개발자 간 협업을 쉽게 함 

**구조**    
기본적으로 요청-응답 구조로 동작    
- 클라이언트가 요청 전송
- 서버가 요청 처리
- 서버가 결과를 응답으로 반환

## 2. REST
(**Re**presentational **S**tate **T**ransfer)   
**REST란?** 웹에서 자원을 효율적으로 관리하기 위한 아키텍처 스타일의 설계 규칙

### 2-1. REST 특징
**자원(Resource) 중심 구조**    
기능이 아닌 데이터(자원) 중심으로 설계
- API 구조 직관적임
- 어떤 데이터인지 URL만 보고 이해 가능

**HTTP 메서드로 의미 구분**     
URL 하나에 기능을 넣지 않고, HTTP 메서드로 행동을 나눔
- URL 깔끔
- 코드 구조 단순화
- 같은 자원도 다양한 동작 가능

**무상태(Stateless)**   
서버는 이전 요청을 저장하지 않음
- 서버 구조 단순
- 여러 서버로 확장 쉬움

**URL은 명사 중심**     
기능이 아닌 자원을 표현하기 때문
- API 구조 표준화
- URL만 봐도 무엇을 다루는지 알 수 있음
- 유지보수 쉬워짐

### 2-2. REST 장점
**구조가 단순하고 이해하기 쉬움**   
복잡한 함수 호출 대신 웹 규치만 사용    
- URL + HTTP 메서드 조합이라 직관적
- 규칙이 일정해서 학습 난이도 낮음

**확장성 좋음**     
서버가 상태를 기억하지 않기 때문
- Stateless 구조 덕분에 서버 여러 개로 확장 가능  
- 트래픽 증가에도 대응 쉬움

**프론트/백엔드 분리 가능**     
개발 속도 증가 + 역할 분리 가능
- 프론트 -> API만 호출
- 백엔드 -> 데이터만 제공

### 2-3. REST 단점
**무상태(Stateless)로 인한 비효율**     
서버가 상태를 기억하지 않음 
- 매 요청마다 인증 정보 포함 필요
- 같은 정보도 반복 전송해야 함

**복잡한 기능 처리 어려움**     
CRUD 중심 구조 -> REST만으로는 설계 불편
- 실시간 채팅
- 복잡한 트랜잭션
- 다중 단계 처리

**설계 품질 차이 발생**  
REST는 강제표준이 아님
- 개발자마다 다르게 설계 가능
- REST처럼 보이는 REST가 아닌 API 존재
