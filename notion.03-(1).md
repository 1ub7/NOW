# Docker
## 1. 개념
**Docker란?** 애플리케이션과 그 실행 환경을 하나로 묶어 어떤 컴퓨터에서도 동일하게 실행할 수 있도록 만드는 컨테이너 기반 플랫폼     
(프로그램이 실행되는 데 필요한 모든 것을 하나의 묶음으로 만들어 어디서든 똑같이 실행하게 하는 기술)     

프로그램만 옮기는 것이 아님     
```
프로그램
+ 라이브러리
+ 실행파일
+ 설정값
+ 필요한 패키지
```
를 다 묶음 (이게 컨테이너)

### 1-1. Docker 등장 배경
기존에는 프로그램을 개발한 뒤 다른 서버로 옮기면 문제가 많았음      
-> 이 문제를 해결하기 위해 만들어짐

**문제가 생겼던 원인**
- 운영체제 차이
- 라이브러리 버전 차이
- 환경 설정 차이
- 의존성 충돌

## 2. 컨테이너(Container)
**컨테이너란?** Docker에서 애플리케이션이 실행되는 독립된 실행 공간    
(프로그램을 다른 프로그램과 분리해서 안전하게 실행시키는 작은 가상 공간)

**왜 필요함?**  
기존에는 여러 프로그램이 하나의 OS에서 직접 실행됨  
-> 같이 설치하면 충돌이 가능했음    
-> 컨테이너 사용 시 서로 영향을 주지 않음

### 2-1. 내부 구성
**애플리케이션(Application)**   
실제로 실행되는 프로그램    
(컨테이너 안에서 중심 역할)
ex) 웹 서버, 파이썬 앱, 데이터베이스 등

**라이브러리(Library)**     
프로그램이 필요로 하는 파일     
(환경 차이를 줄여줌)
ex) python 패키지, node 모듈, 시스템 라이브러리 등

**실행 파일(Binary)**   
프로그램 실행 파일 / 실행 동작 담당

**설정 파일(Configuration)**    
프로그램 동작 설정

**파일 시스템(File System)**    
컨테이너만의 저장 공간

**네트워크 인터페이스**     
외부 통신 담당  
컨테이너마다 고유 IP나 포트 연결 가능함

**프로세스 공간**   
컨테이너는 보통 1개의 메일 프로세스를 기준으로 동작

### 2-2. 특징
**독립성(Isolation)**   
컨테이너는 서로 분리된 공간에서 실행됨  
- 프로세스 분리
- 파일 시스템 분리
- 네트워크 분리     

등의 이유가 있음    
즉, 하나가 문제가 생겨도 다른 컨테이너는 유지될 수 있음.

**경량성(Lightweight)**     
컨테이너는 운영체제를 새로 설치하지 않음    
(앱+필요한 파일만 포함)

**이식성(Protability)**     
같은 컨테이너는 어디서든 동일하게 실행됨
이유 : 환경 자체가 이미지 안에 들어있기 때문

**빠른 실행 속도**  
컨테이너는 OS 부팅 과정이 없음  
(서비스 장애 시 복구가 빠름)

**일관성(Consistency)**     
항상 동일한 환경 유지   
(개발과 운영 차이를 줄임)

### 2-3. 장점
**배포가 매우 쉬움**    
프로그램 설치 필요 없이 
```
이미지 pull
-> 컨테이너 실행
```
즉, 서버 세팅 시간이 줄어듦

**서버 자원 절약**  
여러 컨테이너가 하나의 OS 커널 공유   
- CPU 절약
- RAM 절약
- 저장공간 절약

가능함

**유지보수 편함**   
문제가 생긴다면
```
기존 컨테이너 삭제
-> 새 컨테이너 재생성
```
즉, 복구가 단순함

**확장 쉬움**   
사용자 증가 시 쉽게 늘릴 수 있음
(마이크로서비스 구조에 유리)

**버전 관리 가능**  
이미지마다 버전 지정 가능   
-> 이전 상태로 되돌리기 쉬움

### 2-4. 단점
**보안 취약 가능성**    
호스트 OS 커널 공유
-> 취양점이 있으면 VM보다 완전 격리는 약함  

**데이터 손실 가능성**  
기본적으로 휘발성
-> 삭제하면 내부 데이터 사라질 수 있음

**네트워크 이해 필요**      
컨테이너끼리 통신하려면 설정이 필요함
(초보자한테는 어려움)

## 3. 이미지(Image)
**Docker에서 이미지란?**    
컨테이너를 만들기 위한 설계도   
(컨테이너를 실행하기 전에 미리 준비된 실행 환경 묶음)

**왜 중요함?**      
- 환경 재현 가능
- 배포 간편
- 버전 관리 가능

### 3-1. 역할
컨테이너를 만들기 위한 실행 환경 템플릿 역할    
(컨테이너를 실행하기 전에 준비된 설계도)

**실행 환경 제공**      
프로그램이 실행되기 위한 환경을 담음    
그래서 어느 컨퓨터에서도 같은 환경 실행 가능    

**컨테이너 생성 기준**  
컨테이너는 이미지를 기반으로 만들어짐
즉, 이미지가 없으면 컨테이너 생성 불가

**버전 관리 가능**      
이미지는 버전별 저장 가능   
(이전 상태 복구 가능)

**배포 단순화**     
이미지 하나만 전달하면 됨   
(배포가 쉬워짐)

### 3-2. 이미지 생성 방법
보통 Dockerfile로 만듦   
4번 참고

### 3-3. 계층 구조
**왜 계층 구조를 씀?**  
- 저장 공간 절약    
    공통 레이어 재사용      
    중복 저장 안 함

- 빌드 속도 향상    
    변경된 부분만 다시 생성     
    (코드만 수정할 시 더 빠름)

- 효율적 관리
    레이어 단위 캐시 가능   
    빌드 최적화 가능

## 4. Dockerfile
**Dockerfile이란?** Docker 이미지를 만들기 위한 설정 파일

**왜 필요함?**      
이미지는 Dockerfile에 작성된 내용을 바탕으로 자동 생성됨    
-> 이미지 제작 설계도

**왜 중요함?**  
- 환경을 동일하게 유지할 수 있음
- 반복 작업 자동화
    - 원래는서버마다 직접 설치해야함        
    -> Dockerile이 있으면 자동 처리됨
- 협업이 쉬움
- 유지보수가 편함
    - 버전 관리 가능
    - 누가 어떤 설정 바꿨는지 확인 가능

### 4-1. 기본 흐름
**Dockerfile 작성**     
어떤 환경에서 실행할지 코드로 미리 정의

**Docker build 실행**   
Docker가 Dockerfile을 한 줄씩 읽으면서 각 명령어를 기반으로 이미지 레이어를 생성함

**이미지 생성**     
Docker가 Dockerfile 내용을 하나의 이미지로 저장     
-> 실행 환경 템플릿이 됨

**Docker run**  
생성된 이미지를 기반으로 새로운 컨테이너를 만들라는 명령        
(필요할 시 포트 연결, 볼륨 연결, 환경 변수도 함께 설정)

**컨테이너 실행**   
쓰기 가능한 계층을 추가하고 애플리케이션을 실제로 실행함

### 4-2. 기본 구조
ex
```
FROM python:3.12
WORKDIR /app
COPY . .
RUN pip install -r requirements.txt
EXPOSE 5000
ENV APP_ENV=production
CMD ["python", "app.py"]
```
한 줄씩 문장의 역할을 정의하면?
```
기반 이미지 선택
↓
작업 폴더 지정
↓
파일 복사
↓
필요한 설치
↓
포트 지정
↓
환경 변수 설정
↓
실행 명령 지정
```

### 4-3. 주요 명령어
**FROM**    
이미지를 만들 때 어떤 운영 환경을 사용할지 정함     
Docker 이미지는 반드시!! 하나의 기반 이미지 위에서 시작     
-> 운영체제나 언어 런타임이 포함된 이미지를 선택

**WORKDIR**     
컨테이너 내부에서 앞으로 작업할 폴더 설정   
이후 명령어는 해당 경로를 기준으로 실행
- 경로를 반복해서 쓰지 않아도 됨
- 파일 관리가 편함
- 실행 위치 통일

등의 장점이 있음

**COPY**    
현재 폴더 파일을 이미지 내부로 복사     
프로그램 코드나 설정 파일을 컨테이너 안으로 넣을 때 사용    

**RUN**     
이미지 생성 중 필요한 명령을 실행       
컨테이너가 실행될 때가 아니라 빌드할 때 실행됨  
주로 하는 일
- 패키지 설치
- 라이브러리 설치
- 업데이트
- 설정 변경

**EXPOSE**  
컨테이너가 사용할 포트 명시     
외부에서 어떤 포트로 접근할지 알려줌
주의할점!!! 포트를 여는게 아니라 포트 정보를 명시하는 역할

**ENV**     
컨테이너 안에서 사용할 환경 변수 저장    
프로그램 설정값을 코드 밖에서 관리할 수 있음    
(프로그램은 이걸 읽어 동작 변경 가능)

**CMD**     
컨테이너 시작 시 실행할 기본 명령   
(즉, 프로그램 시작 담당)    
-> CMD는 기본값이라서 실행 시 변경 가능

## 5. 컨테이너와 이미지 차이점
| 구분 | 이미지 | 컨테이너 |
| ----- | ----- | ----- |
| 의미 | 실행 환경이 담긴 설계도 | 이미지를 실행한 실제 상태  |
| 상태 | 읽기 전용 | 쓰기 가능 |
| 역할 | 컨테이너 생성 기준 | 프로그램 실제 실행 |
| 변경 여부 | 직접 변경되지 않음 | 실행 중 변경 가능 |
| 개수 | 하나로 여러 개 생성 가능 | 이미지마다 여러 개 가능 |

## 6. Docker 하는 일
**애플리케이션을 컨테이너로 패키징**    
프로그램 실행에 필요한 파일과 설정을 하나의 컨테이너 이미지로 묶음      
-> 개발 환경과 운영 환경이 동일해져 환경 차이로 발생하는 오류를 줄일 수 있음   

**컨테이너 실행**   
패키징된 이미지를 기반으로 독립된 실행 공간을 만듦      
-> 복잡한 설치 과정 없이 명령어 하나로 빠르게 실행      
-> 운영 효율 높아짐

**실행 환경 통일**  
운영체제나 라이브러리 버전 차이로 인한 문제를 줄임      
(같은 이미지 사용하므로 동일한 동작 결과 보장)

**프로그램 및 이미지 관리**     
프로그램을 서로 분리항 독립적으로 실행할 수 있음    

**배포 자동화**     
이미지를 저장소에 올린 뒤 필요한 서벙에서 바로 내려받아 실행 가능   
->CI/CD 환경과 연결하면 코드 변경 후 자동 빌드와 자동 배포까지 가능

**리소스 관리**     
각 컨테이너마다 CPU와 메모리 사용량을 제한할 수 있음    
특정 컨테이너가 서버 자원을 과도하게 사용하는 것을 방지해 안정성을 높임

## 7. 주요 구성 요소
| 구성 요소 | 역할 |
| ----- | ----- |
| Client | 사용자 명령 입력 |
| Host | 실행 환경 |
| Daemon | 핵심 관리 |
| Image | 설계도 |
| Container | 실행 결과 |
| Registry | 이미지 저장소 |
| Network | 통신 |
| Volume | 데이터 저장 |

## 8. 동작 원리
**사용자 명령 입력**        
사용자가 터미널에서 Docker 명령 입력    
↓   
**Docker Client 전달**      
Client는 명령 해석 / Docker 엔진에 전달함   
↓   
**Docker Daemon 처리**  
명령이 Daemon에 전달됨      
-> Daemon은 Docker의 핵심 엔진임    
↓   
**이미지 존재 여부 확인**     
Daemon이 먼저 이미지가 있는지 검사      
↓   
**컨테이너 생성**   
이미지 기반으로 컨테이너 생성   
↓   
**네트워크 연결**   
생성된 컨테이너가 외부 또는 다른 컨테이너랑 통신할 수 있도록 연결   
↓   
**프로세스 실행**   
컨테이너 내부에서 지정된 애플리케이션 프로세스를 실행함     

## 9. 특징
**컨테이너 기반 가상화**    
운영체제 전체를 만드는 것이 아닌 애플리케이션만 독립된 공간에서 실행    

**환경 일관성**     
개발 환경과 운영 환경을 동일하게 유지함     

**빠른 실행 속도**  
OS 전체 부팅이 없어서 빠름  

**이미지 기반 관리**   
이미지라는 템플릿으로 관리함

## 6. 장점
**배포가 매우 쉬움**    
이미지 하나만 전달하면 됨   
(즉, 배포 속도 향상)

**서버 자원 절약**      
커널 공유 구조라 메모리 사용량 적음     

**유지보수 편리**   
문제 생기면 삭제 후 재생성 가능     
-> 복구 빠름

**확장 쉬움**   
트래픽 증가 시 컨테이너 여러 개 추가 가능   
(클라우드 환경에 적합)

## 7. 단점
**보안 문제 가능성**    
호스트 커널 공유    

**데이터 관리 어려움**  
컨테이너 삭제 시 데이터 손실 가능

**운영 복잡성 증가**    
컨테이너 많아지면 관리 어려움

**OS 의존성**   
리눅스에 최적화     
(Windows에서는 제한 가능)
